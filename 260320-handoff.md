# 交接文档 - 微信小程序体验对齐 H5

> 日期: 2026-03-20
> 项目: touzi
> 本轮目标: 按 H5 版本体验重做微信小程序首页的视觉与交互表现

## 1. 当前任务目标

当前要解决的问题是：微信小程序版本虽然已具备基本功能，但用户明确反馈“体验特别差”，与 H5 版本差距很大，尤其是骰子的立体感、动画自然度、背景质感和整体沉浸感不足。

预期产出：
- 微信小程序首页在视觉语言和交互节奏上尽量贴近 H5 版本
- 保留现有小程序能力：摇骰子、设置面板、音效、屏幕常亮、骰子数量调节
- 在 Canvas 2D 方案下尽量提升“像 H5”的程度，而不是追求真 3D 引擎

完成标准：
- 点击“摇”后能看到接近 H5 的多面立方体翻转、抛起和收束动画
- 骰子材质更接近 H5：象牙白、边缘高光、凹陷点数、立体阴影
- 背景更接近 H5：稳定的深绿色毛毡质感，避免每帧闪烁噪点
- UI 控件层级、按钮质感、设置弹窗风格更接近 H5
- 最终停面与点数结果一致
- 1-10 个骰子下不应明显重叠

## 3. 关键上下文

重要背景信息：
- H5 版本基于 Three.js，微信小程序版本当前仍采用 Canvas 2D
- 上一份交接 `260319-handoff.md` 已明确：XR-Frame 曾尝试但体验/兼容性不理想，因此当前继续走 Canvas 2D 路线
- 用户对 H5 版本体验“很满意”，对小程序版本“特别差”，要求按 H5 体验修改小程序

用户明确要求：
- 不要陀螺仪/摇一摇
- 不要“摇出时间”显示
- 竖屏 only
- 骰子数量 1-10 个，默认 5 个
- 重点是体验对齐 H5，不是只保证功能可用

已知约束：
- 小程序使用 `canvas type="2d"`
- 小程序没有浏览器版 `requestAnimationFrame`，当前仍使用 `setTimeout(..., 16)`
- `wx.createOffscreenCanvas` 可能存在兼容性风险，所以代码中保留了 fallback 思路
- 当前环境不能直接运行微信开发者工具和真机调试

已做出的关键决定：
- 不重新启用 XR-Frame
- 不引入 Three.js 到小程序
- 不再沿用旧版“只贴正面纹理 + 拉伸”的绘制方式
- 采用“2D Canvas 中逐面绘制可见立方体面片”的折中方案，目标是最大程度模拟 H5 的空间感

重要假设：
- 现阶段用户最在意的是“视觉像不像 H5”和“动画手感像不像 H5”，而不是底层技术是否完全一致
- 在 Canvas 2D 的限制下，若把面片透视、材质、阴影、背景和 UI 同时做好，可以显著缩小与 H5 的主观差距

## 4. 关键发现

1. 之前小程序版本体验差，不只是“3D 不够真”，而是多个问题叠加：
- 骰子仅绘制单一正面贴图，导致空间感极弱
- 背景毛毡纹理是每帧随机生成，产生明显闪烁感
- 阴影、深度排序、停面朝向都偏粗糙
- 按钮和设置面板风格也没有真正贴近 H5

2. 本轮最关键的实现改动在 `pages/index/index.js`：
- 文件顶部定义了 `FACE_VALUES`、`FACE_DEFINITIONS`、`LIGHTS`
- 核心渲染链路是：
  - `draw()`
  - `drawDice3D()`
  - `getVisibleFaces()`
  - `renderFace()`
  - `rotateVector()`
- 这是当前小程序体验提升的核心，不要轻易回退到旧版画法

3. 骰子最终停面现在通过 `getRotationForValue()` 控制：
- 1 -> `{ x: 0, y: 0, z: 0 }`
- 6 -> `{ x: 180, y: 0, z: 0 }`
- 2 -> `{ x: -90, y: 0, z: 0 }`
- 5 -> `{ x: 90, y: 0, z: 0 }`
- 3 -> `{ x: 0, y: -90, z: 0 }`
- 4 -> `{ x: 0, y: 90, z: 0 }`

4. 背景闪烁问题已从根上处理：
- 旧逻辑在 `drawBackground()` 中每帧随机撒点
- 新逻辑通过 `generateBackgroundTexture()` 预生成纹理缓存，再平铺绘制

5. 目前最可能还需要继续调的不是功能，而是“参数”：
- `generateNonOverlappingPositions()` 中的散布范围和最小间距
- `drawDice3D()` / `getVisibleFaces()` 中的透视和尺寸缩放
- `prepareRoll()` / `animate()` 中的旋转圈数、bounce 高度、收束时机

6. 当前没有做真机视觉校准，因此还不能确认：
- 可见面投影是否在小程序 Canvas 上表现稳定
- `ctx.transform(...)` + `clip()` 的组合在开发者工具和真机上是否完全一致
- 高 DPI 下贴图边缘是否会发虚

## 7. 风险与注意事项

1. 不要误判为“只要功能正常就算完成”
- 用户当前真正不满的是体验，尤其是主观视觉差距

2. 不要轻易把当前逐面绘制方案回退成旧版单贴图方案
- 旧版就是导致体验差的主要原因之一

3. 不要重复探索 XR-Frame 或小程序 Three.js
- 之前交接文档已经说明，这条路线已基本判定不值得继续

4. 需要警惕小程序 Canvas 的兼容性差异
- `OffscreenCanvas`
- `clip()`
- `transform()`
- SVG `image` 的渲染

5. `git diff` 当前未能直接反映本轮修改
- 本轮是通过删除再新增文件的方式重写 `pages/index/index.js` 和 `pages/index/index.wxss`
- 如需确认，请直接打开文件内容，不要依赖空的 diff 输出误判“没有改动”

6. 当前未做性能评估
- 如果多骰子时掉帧，优先降低贴图细节和背景纹理密度，不要先砍掉核心立体绘制逻辑

7. `resolveOverlaps()` 仍是近似解
- 目前只是避免明显重叠，不保证分布最自然
- 若用户后续继续指出“落点假”或“推开不自然”，这是优先可继续优化的点

## 下一位 Agent 的第一步建议

1. 先打开 `pages/index/index.js`，快速确认当前已经不是旧版单贴图渲染，而是“逐面绘制可见面”的实现。
2. 立刻在微信开发者工具中编译首页，连续摇 10-20 次，对照 H5 看最明显的差距还剩什么。
3. 如果没有兼容性错误，优先只调 4 组参数：
- `generateNonOverlappingPositions()` 的 `spreadX / spreadY / minDistance`
- `drawDice3D()` 的阴影位置和透明度
- `getVisibleFaces()` 的 `size` 深度系数
- `animate()` 的 `bounce` 高度和两个阶段的时间比例
4. 只有在当前方案真机表现不稳定时，才考虑加降级渲染路径；不要先重构。
