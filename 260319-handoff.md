- 骰子初始位置和最终位置都不重叠

## 3. 关键上下文

### 技术方案选择
- **渲染方案**: Canvas 2D（而非 XR-Frame），因为 XR-Frame 在小程序中配置复杂且真机兼容性差
- **骰子绘制**: 使用 `OffscreenCanvas` 预生成骰子各面纹理（256x256 像素），然后在主 Canvas 上绘制
- **3D 效果**: 通过 2D Canvas 的 `scale`, `rotate`, `translate` 模拟透视和旋转

### H5 参考实现
- H5 使用 Three.js，`js/dice.js` 中用 `BoxGeometry` + `CanvasTexture` 生成骰子
- H5 防重叠算法核心参数：`minDistance = diceSize * 1.3`（约 1.6 单位）
- H5 动画缓动：`easeProgress = 1 - Math.pow(1 - progress, 3)`
- H5 光照：环境光 + 方向光 + 点光 + 轮廓光

## 4. 关键发现

### 当前实现的主要差距
1. **视觉效果差距**: 
   - Canvas 2D 模拟的 3D 效果与 Three.js 真实 3D 有本质差距
   - 光照、阴影、材质质感不如 H5 自然
   - 骰子缺乏"厚重感"和"立体感"

2. **动画差距**:
   - 弹跳高度和时机可能与 H5 不一致
   - 旋转的多轴向组合效果不如 H5 流畅

3. **防重叠**:
   - 初始位置生成已实现不重叠
   - 动画结束后的 `resolveOverlaps()` 会强制推开重叠骰子，可能导致位置不自然

### 技术限制
- 小程序 Canvas 2D 的 `OffscreenCanvas` 在某些机型上可能不支持
- 小程序没有 `requestAnimationFrame`，使用 `setTimeout(fn, 16)` 代替
- 真机调试时 Canvas 性能可能不如浏览器

### 需要改进的方向
- 骰子纹理：当前使用程序生成，可能需要更精细的绘制
- 3D 透视：当前的透视模拟可能过于简化
- 光照效果：可能需要更复杂的光照计算

## 7. 风险与注意事项

### 已验证（不建议重复尝试）
- ✅ XR-Frame 配置复杂，真机兼容性差，不建议再用
- ✅ Canvas 2D 可以实现基本功能，但 3D 效果有天花板

### 易踩坑点
- ⚠️ `OffscreenCanvas` 在部分机型上不支持，需要 fallback
- ⚠️ 小程序 Canvas 的 `drawImage` 性能可能不如浏览器
- ⚠️ 旋转角度累积会导致透视计算溢出，需要取模

### 不要做的方向
- ❌ 不要尝试在小程序中使用 Three.js（包体积过大）
- ❌ 不要添加陀螺仪功能（用户明确不要）
- ❌ 不要添加"摇出时间"显示（用户已要求移除）

### 关键决策点
- 是否继续优化 Canvas 2D，还是换其他方案？
- 是否可以接受"接近 H5 效果"而非"完全一致"？
